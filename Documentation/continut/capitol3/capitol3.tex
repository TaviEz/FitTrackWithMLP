\section{System requirements and High-Level Design}

\subsection{Problem Reiteration and Solution Strategy}

In the current digital health landscape, nutritional tracking applications frequently fail 
to sustain long-term user adherence due to structural rigidity. Traditional platforms 
rely on manual, repetitive data logging where users must explicitly search, weigh, and 
log every individual food item consumed. Furthermore, generic meal planning engines often 
deliver static, pre-packaged templates that fail to account for unique metabolic shifts, 
individual lifestyle activity factors, or lean muscle mass preservation requirements.

The strategy proposed in this thesis addresses these core limitations by shifting the 
paradigm from manual tracking to automated macro-nutrient synthesis. This is achieved 
through a decentralized microservice architecture that decouples user identity tracking 
from mathematical plan generation. The solution introduces an autonomous optimization 
engine, the OptimizerService, which dynamically calculates nutritional targets and 
automatically compiles meal configurations tailored to the user's specific metabolic 
profile. By automating the data-entry layer and providing instant, scientifically backend 
plan variations, the platform eliminates the traditional points of friction in dietary 
logging while maintaining high structural flexibility.


\subsection{Original Ideas and Innovative Solutions}
\begin{itemize}
    \item \textbf{Embedded Read-Only Database for Meal Optimization:} The system separates data 
    storage concerns across two distinct database engines based on their access patterns. 
    Transactional data such as user profiles, authentication state, and daily plans are persisted 
    in a cloud-hosted Azure SQL instance. The \texttt{OptimizerService}, which handles meal 
    generation, operates exclusively on static pre-seeded nutritional data that never changes at 
    runtime. For this reason, an embedded SQLite database is deployed directly alongside the service 
    container, eliminating network round-trips to a cloud database for read-only lookups during meal 
    plan generation. Additionally, the \texttt{DailyPlanService} maintains local snapshots of this 
    nutritional data, removing any runtime dependency on the \texttt{OptimizerService}'s internal storage.

    \item \textbf{Dynamic Lean Mass Preservation Target Calibration:} Unlike typical platforms that 
    apply rigid calorie deficits across all profiles, this solution implements a dynamically scaled 
    allocation algorithm within the \texttt{DailyPlanService}. The system evaluates the user's 
    active configuration against specific constraints using the following relationship:
    
    \begin{equation}
        TDEE = BMR \times \text{ActivityFactor}
    \end{equation}
    
    When a restrictive deficit is selected, the system automatically recalibrates macro-nutrient 
    distributions, scaling protein allocations up to 1.4g/kg or 2.0g/kg based on calculated lean 
    mass preservation metrics while strictly enforcing a minimum fat threshold of 0.7g/kg or 0.8g/kg 
    to protect metabolic health. This separation ensures nutritional logic remains within the .NET 
    domain layer, keeping the \texttt{OptimizerService} focused solely on meal selection.

    \item \textbf{Stateless Authentication with Refresh Token Rotation:} The system implements a
    dual-token authentication strategy to balance security and user experience across a 
    cross-domain microservices architecture. Short-lived JWT access tokens are stored 
    exclusively in JavaScript memory rather than cookies, eliminating CSRF attack vectors 
    entirely. Long-lived refresh tokens are transmitted via HttpOnly cookies, making them 
    inaccessible to JavaScript and resistant to XSS attacks. Upon each refresh, the existing 
    token is invalidated and replaced, ensuring a stolen token can only be used once before 
    being revoked.
\end{itemize}

\section{User Requirements}

The application's scope is driven by the practical needs of the end user, divided into functional 
operations and non-functional quality expectations.

\subsection{Functional Requirements}
\begin{itemize}
    \item \textbf{Authentication:} Users must be able to sign up, log in securely, and 
    remain authenticated across page refreshes.
    \item \textbf{Onboarding Form:} First-time users require a guided form to enter basic 
    metrics (weight, height, age, sex, and activity level).
    \item \textbf{Metabolic Calculation:} The system must automatically compute and display 
    the user's BMR and TDEE based on their onboarding input.
    \item \textbf{Diet Goal Customization:} Users need to select a specific goal 
    (e.g., Lean Cut, Lean Bulk, or Lose Fat) which automatically scales their calorie and 
    macro targets.
    \item \textbf{Meal Plan Selection:} Users must be able to choose between automatically 
    generating a meal plan via the AI optimizer or manually adding foods.
    \item \textbf{Dashboard CRUD:} The main interface must allow users to view their daily 
    macro progress, log consumed food items, update metrics, and delete entries.
\end{itemize}

\subsection{Non-Functional Requirements}
\begin{itemize}
    \item \textbf{Performance:} Page transitions, database saves, and macro calculations 
    must happen within a sub-second timeframe to ensure a smooth user experience.
    \item \textbf{Data Privacy:} User profiles, target metrics, and logged food records must 
    be securely isolated so users can only access their own data.
\end{itemize}

\section{Architectural Modeling}

\subsection{System Architecture}

The system is designed using a decentralized microservice architecture to ensure 
scalability, fault isolation, and strict separation of business concerns. Each service 
operates independently with its own data store, exposing functionality exclusively through 
well-defined HTTP APIs. The structural organization of the system's components, their 
interfaces, and data boundaries are illustrated in the component diagram below.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.9\textwidth]{continut/capitol3/figuri/ComponentDiagram.pdf}
    \caption{System Component Architecture Diagram}
    \label{fig:component_architecture}
\end{figure}

As shown in Figure~\ref{fig:component_architecture}, the system architecture is composed of 
the following core layers and components:

\begin{itemize}
    \item \textbf{Presentation Layer (React SPA):} A lightweight, client-side single-page 
    application built with React. It handles user interactions, manages local routing states, 
    and communicates asynchronously with the backend APIs via standard RESTful HTTP protocols 
    using JSON payloads.
    \item \textbf{User Management Service (\texttt{UserManagementService}):} A specialized 
    backend service responsible for user administration. It processes user registration, 
    validates credentials during login, handles secure session management, and stores physical 
    user profile metrics.
    \item \textbf{Daily Plan Service (\texttt{DailyPlanService}):} The orchestrator for 
    the user's active nutritional tracking. It handles core CRUD operations related to food
    logging, manages daily macro-nutrient targets based on selected goals, and coordinates
    meal plan persistence. When automated meal generation is requested, it delegates to the
    \texttt{OptimizerService} via a typed \texttt{HttpClient}, forwarding the user's 
    targets and receiving back an optimized meal configuration.
    \item \textbf{Optimizer Service (\texttt{OptimizerService}):} A highly isolated 
    computational engine built exclusively for running the meal generation algorithms. It 
    ingests targeted macro goals and selects optimal food combinations from its embedded 
    storage without interfering with transactional user flows.
    \item \textbf{Data Tier Topology:} Storage concerns are explicitly segregated. Relational 
    transactional records and user authorization states are persisted in the 
    cloud-hosted \texttt{AuthServiceDb} (Azure SQL). In contrast, the static food and 
    ingredient catalogs used for macro synthesis are hosted within an embedded SQLite 
    database running locally inside the \texttt{OptimizerService} container to minimize 
    network latency.
\end{itemize}

\subsection{Database Design}

The system employs three distinct databases, each scoped to its respective service to 
ensure strict data isolation and separation of concerns.

\subsubsection{UserManagement Service Database}
The \textit{AuthServiceDb} manages user identity, authentication state, and physical profile 
data. The \textit{AspNetUsers} table is provided by the ASP.NET Core Identity framework and 
serves as the central user entity. The \textit{UserDetails} table maintains a one-to-one 
relationship with \textit{AspNetUsers}, storing user physiological data used for 
nutritional calculations. The \textit{RefreshTokens} table maintains a one-to-many 
relationship with \textit{AspNetUsers}, persisting refresh tokens issued after successful 
authentication to support token rotation.

\section{Implementation Description}
This section describes the primary classes and programming components 
developed to implement the platform's core business logic. The backend 
architecture follows a clean, layered structure where controllers expose 
the API endpoints, services handle business operations, and entities 
model the core data structures.

\subsection{Description of Implemented Elements and Classes}

Both backend services follow a uniform three-layer structure: a controller at the API
boundary, a service class containing the domain logic, and a \texttt{DbContext} injected
directly into the service for data access. This pattern is applied consistently across
all features, with each layer depending on its neighbour exclusively through an interface,
enabling independent testability and clear separation of concerns. The components described
below represent the most significant classes implemented within this structure.

\begin{itemize}
    \item \textbf{\texttt{UserController}:} Positioned within the API exposure
    layer of the \texttt{UserManagementService}. It exposes the complete HTTP
    interface for session and profile lifecycle management, covering both
    anonymous operations such as registration, login, and token refresh, and
    authenticated operations such as logout, identity retrieval, and physical
    metric persistence. Security boundaries are enforced at the controller level
    using the \texttt{[Authorize]} attribute on all routes that require a valid
    access token.
\end{itemize}

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.9\textwidth]{continut/capitol3/figuri/UMLtemplateDiagram.pdf}
    \caption{UML template diagram}
    \label{fig:uml}
\end{figure}

\begin{itemize}
    \item \textbf{\texttt{UserDetailsService}:} Implements the core domain logic for user 
    processing. It coordinates payload validation, computes basic metabolic baselines 
    from raw physical data, and triggers persistent repository writes to the database 
    tier.
    
    \item \textbf{Token Infrastructure (\texttt{TokenService} \& \texttt{RefreshTokenService}):}
    Together these two components implement the dual-token authentication strategy.
    \texttt{TokenService} handles the cryptographic generation of short-lived JWT access
    tokens, signing them with the shared secret and embedding the user's identity claims.
    \texttt{RefreshTokenService} manages the lifecycle of long-lived session tokens,
    persisting them to the database on issuance, validating them on each refresh request,
    and deleting them on logout to enforce single-use rotation.

    \item \textbf{\texttt{DailyPlanController}:} Exposes the API endpoints for managing 
    tracking records and triggering menu synthesis. It handles the integration between 
    the user front-end and the optimization core through two key 
    operations: \texttt{POST /api/dailyplan/generate}, which redirects user physical 
    parameters to the internal optimization engine to build a macro-calibrated menu, 
    and \texttt{GET /api/dailyplan/ingredient/search}, which queries available food 
    data pools. Additionally, it implements core CRUD operations across both the meal 
    and ingredient levels, specifically allowing users to fetch complete plans, modify 
    specific meal titles, and add, update, or delete individual ingredients within a 
    planned meal slot.
    
    \item \textbf{\texttt{DailyPlanService}:} Implements the \texttt{IDailyPlanService} 
    interface to execute the business logic behind the active tracking dashboard. 
    It acts as the direct coordinator between the data repositories and the controller 
    layer, handling state modifications across both meal and ingredient scopes. It 
    processes date-specific plan lookups (\texttt{GetDailyPlanAsync}), manages full 
    layout initializations or structural replacements, and evaluates modification rules 
    whenever users add, update, or remove ingredients and meal slot parameters.
    
    \item \textbf{\texttt{MealOptimizerClient}:} Implements the
    \texttt{IMealOptimizerClient} interface to abstract the communication with
    the underlying optimization engine service. It provides a clean, async gateway
    to forward user metabolic configurations and macro-nutrient targets down to the
    solver (\texttt{OptimizeAsync}) and also forwards the keyword-matching
    logic (\texttt{GetIngredientOptionsAsync}) to populate available food selections
    during custom plan creation.

\end{itemize}

\subsection{Algorithm Pseudocode}

The meal selection algorithm generates a complete daily plan by distributing nutritional 
targets across four meal categories using fixed caloric split ratios. For each category, 
a pre-trained MLP model predicts the top $k$ most suitable meals based on calorie and 
protein targets. The algorithm filters suggestions by complexity level and excludes 
previously consumed meals, falling back to the top model suggestion if no match is found. 
Each selected meal is then passed to the ingredient optimization algorithm to calibrate 
portion sizes.

\begin{algorithm}
\caption{Meal Selection Algorithm}
\begin{algorithmic}[1]
\Require goals (calories, protein, min\_fat), target\_complexity, excluded\_meal\_ids
\Ensure daily\_plan list of 4 meals

\State splits $\gets$ \{BREAKFAST: 0.25, LUNCH: 0.30, SNACK: 0.15, DINNER: 0.30\}
\State daily\_plan $\gets$ []

\For{each meal\_category in [BREAKFAST, LUNCH, SNACK, DINNER]}
    \State ratio $\gets$ splits[meal\_category]
    \State meal\_targets $\gets$ goals $\times$ ratio

    \State k $\gets$ 15 if target\_complexity = "Simple" else 5
    \State top\_indices $\gets$ MLP.predict(meal\_targets.calories, meal\_targets.protein, meal\_category, k)
    \State suggested\_ids $\gets$ map top\_indices to actual meal IDs

    \State final\_meal $\gets$ null
    \For{each meal\_id in suggested\_ids}
        \If{meal\_id $\in$ excluded\_meal\_ids} \textbf{continue} \EndIf
        \State meal $\gets$ MealRepository.get(meal\_id)
        \If{meal.complexity = target\_complexity \textbf{and} meal.category = meal\_category}
            \State final\_meal $\gets$ meal
            \State \textbf{break}
        \EndIf
    \EndFor

    \If{final\_meal = null}
        \State final\_meal $\gets$ MealRepository.get(suggested\_ids[0]) \Comment{Fallback}
    \EndIf

    \State result $\gets$ MealOptimizer.optimize(final\_meal.ingredients, meal\_targets)

    \State daily\_plan.append(result)
\EndFor

\State \Return daily\_plan
\end{algorithmic}
\end{algorithm}

The ingredient optimization algorithm determines the optimal portion size in grams for 
each ingredient in a selected meal. Using Sequential Least Squares Programming (SLSQP), 
it minimizes a weighted squared error objective function over calorie and protein 
targets, while enforcing a minimum fat constraint to protect metabolic health. Each 
ingredient is bounded by category-specific minimum and maximum gram values to ensure 
realistic portion sizes.

\clearpage
\begin{algorithm}
\caption{Ingredient Weight Optimization via SLSQP}
\begin{algorithmic}[1]
\Require ingredients list, targets (calories, protein, min\_fat)
\Ensure optimized ingredient weights in grams

\State data\_matrix $\gets$ nutrient values per 1g for each ingredient
\State bounds $\gets$ (min\_g, max\_g) per ingredient based on food category
\State initial\_guess $\gets$ reasonable starting portion per ingredient category

\Function{Objective}{weights}
    \State current\_cal $\gets$ weights $\cdot$ data\_matrix[:, calories]
    \State current\_protein $\gets$ weights $\cdot$ data\_matrix[:, protein]
    \State cal\_error $\gets$ $(current\_cal - target\_cal)^2$
    \State protein\_error $\gets$ $(current\_protein - target\_protein)^2$
    \State \Return $cal\_error + 20 \times protein\_error$ \Comment{protein weighted higher for macro accuracy}
\EndFunction

\State constraints $\gets$ \{min\_fat: weights $\cdot$ data\_matrix[:, fat] $\geq$ targets.min\_fat\}

\State result $\gets$ SLSQP.minimize(Objective, initial\_guess, bounds, constraints)

\If{result.success}
    \State \Return optimized ingredient list with weights
\Else
    \State \Return null
\EndIf

\end{algorithmic}
\end{algorithm}


\section{Technology Stack Selection and Justification}

The selection of the technology stack was driven by the computational demands of the 
meal optimization algorithm, the need for a highly responsive user experience, and the 
requirement for robust security boundaries within a microservices architecture. 

\subsection{Backend Framework: ASP.NET Core (.NET 10)}
ASP.NET Core was selected as the foundational framework for the microservices backend due 
to several architectural advantages:
\begin{itemize}
    \item \textbf{High Throughput and Performance:} The .NET services act as API
    orchestrators handling concurrent user sessions, async database reads and writes,
    and inter-service HTTP calls to the optimizer. ASP.NET Core's Kestrel server is
    purpose-built for high-concurrency I/O-bound workloads, making it well-suited for
    this coordination role without introducing unnecessary processing overhead.
    \item \textbf{Asynchronous Programming Model:} The native async/await patterns
    keep API endpoints non-blocking during database queries and outbound HTTP calls,
    freeing threads to serve other requests and maintaining responsiveness under
    concurrent load.
    \item \textbf{Built-in Dependency Injection:} The framework natively supports 
    clean dependency inversion interfaces, which facilitates decoupling controllers 
    from core business service handlers.
\end{itemize}

\subsection{Frontend Library: React (TypeScript)}
The user interface is constructed as a client-side Single Page Application (SPA) 
using React, engineered to minimize user tracking friction:
\begin{itemize}
    \item \textbf{Component-Based Architecture:} It enables isolation of UI components, 
    allowing the daily dashboard progress tracking macros, the onboarding setup wizards, 
    and individual ingredient search logs to render dynamically.
    \item \textbf{Virtual DOM and Sub-Second Updates:} React ensures that modifications 
    to meal logs, macro-nutrient calculations, or target updates reflect immediately 
    on-screen without requiring complete page refreshes.
    \item \textbf{Strict Typing with TypeScript:} TypeScript was layered over React to 
    enforce strict data contracts, ensuring data mapping to backend Data Transfer 
    Objects (DTOs) remains reliable.
\end{itemize}

\subsection{Data Management Tier: Azure SQL and SQLite}
To eliminate standard cloud infrastructure network bottlenecks, a hybrid database 
approach was chosen:
\begin{itemize}
    \item \textbf{Azure SQL (Relational Engine):} Used for cloud storage of relational 
    user transactional records, authentication security states, and persistent historical 
    tracking plans. It guarantees strict ACID compliance and secure multi-tenant isolation 
    boundaries.
    \item \textbf{Embedded SQLite:} Deployed directly alongside the execution container 
    of the \texttt{OptimizerService}. Because the ingredient pool consists of static, 
    pre-seeded nutritional profiles, storing this data in an embedded local file 
    eliminates cloud database network round-trip overhead entirely, allowing sub-second 
    lookups during calculation loops.
\end{itemize}

\subsection{Hardware Platform Analysis}

The application runs entirely on Azure cloud infrastructure, with no dependency on
physical server management. Each of the three microservices is packaged as a
Docker image, built locally, and pushed to the \textbf{Azure Container Registry (ACR)},
which serves as the central image repository for the deployment pipeline. The containers
are then deployed as independent instances within the \textbf{Azure Container environment},
where each service receives its own isolated compute allocation.

The two ASP.NET Core services are predominantly I/O-bound, spending the majority of
their execution time waiting on database queries and outbound HTTP calls rather than
performing intensive computation. The Python \texttt{OptimizerService} is the only
CPU-intensive workload in the system, performing numerical optimization during meal
plan generation. This difference in resource profiles means the two service types
could be scaled independently based on demand without affecting each other.

The transactional database tier is provided by \textbf{Azure SQL}, a fully managed
relational database service that handles availability, automated backups, and failover
at the platform level. Because the database is managed externally to the containers,
it remains persistent and unaffected by container restarts or redeployments.

This cloud-native deployment model also ensures environment parity between local
development, where services run via Docker Compose from the same images, and
production, where those images are pulled from ACR and executed in the Azure
container environment.

\subsection{Advantages and Disadvantages}

The microservice architecture and cloud-native deployment model offer several structural
benefits, but also introduce trade-offs that are worth acknowledging.

\textbf{Advantages:}
\begin{itemize}
    \item \textbf{Fault Isolation:} A failure or restart in the \texttt{OptimizerService}
    does not affect the \newline\texttt{UserManagementService} or the daily plan 
    tracking flow. Each container operates independently, limiting the blast radius of 
    any single service degradation.

    \item \textbf{Independent Deployability:} Each service can be rebuilt, pushed to ACR,
    and redeployed without touching the others. The Python optimization engine, for
    instance, can receive algorithm updates independently of the .NET authentication layer.

    \item \textbf{Data Ownership and Schema Isolation:} Each service controls its own
    database schema. There is no shared database that could create hidden coupling between
    services, making schema migrations safer and more contained.

    \item \textbf{Technology Heterogeneity:} The architecture allowed the optimization
    engine to be implemented in Python, which provides access to the scientific computing
    ecosystem (scipy, scikit-learn), while the API layer benefits from the performance and
    type safety of .NET. Each service uses the language best suited to its role.
\end{itemize}

\textbf{Disadvantages:}
\begin{itemize}
    \item \textbf{Operational Complexity:} Running three containers alongside two database
    instances locally requires Docker Compose coordination. Onboarding to the development
    environment is more involved than a single-process monolith.

    \item \textbf{Inter-Service Network Overhead:} The meal plan generation flow requires
    an HTTP call from the \texttt{DailyPlanService} to the \texttt{OptimizerService},
    introducing network latency that would not exist in an in-process call within a
    monolithic architecture.

    \item \textbf{Distributed Debugging:} Tracing a failure that spans multiple services
    requires correlating logs across separate containers, which is more demanding than
    following a single stack trace in a unified process.

    \item \textbf{Cold Start Latency:} Azure container instances that scale to zero
    will incur a startup delay on the first incoming request. For the
    \texttt{OptimizerService}, which loads an MLP model on startup, this cold start
    period is more pronounced than for a stateless HTTP service.
\end{itemize}
